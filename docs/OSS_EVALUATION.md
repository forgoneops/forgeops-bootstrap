# OSS Candidate Evaluation

Research-backed evaluation of every OSS project considered for the ForgeOps platform layers (see `PROJECT_SPEC.md`'s "Platform Vision" section for the layer definitions). This is a living document — update the verdict for a candidate whenever new information changes it, and add new candidates as they come up. Findings below are from a single research pass and should be re-verified (star count, maintenance activity, licensing) before actually integrating anything, since these numbers age quickly.

## Cross-cutting notes

- **vexp and Context Mode** solve overlapping problems (agent context/token compression) to each other, and partially to the fragmented "Claude Token Optimizer" space — evaluate them side by side rather than in isolation, and confirm vexp's actual OSS licensing before treating it as self-hostable OSS.
- **Graphiti, Letta, Cognee, Mem0** all compete for the Memory/Knowledge layer — running all four would be duplicative. Pick one primary per layer; the others get documented as rejected-as-redundant, not silently dropped.
- **Continue and Cline** are clients, not backend integration targets — they belong in the Clients row of the architecture table, not evaluated as things ForgeOps integrates into its own layers. Continue's archived status makes it a weak long-term client bet versus Cline's active growth.
- **Bifrost (LLM-provider gateway) and the MCP gateway (`mcp-context-forge`/`docker/mcp-gateway`)** are two different gateway concerns — one routes to LLM providers, the other routes to MCP tool servers. Both may eventually be needed; they are not the same decision.
