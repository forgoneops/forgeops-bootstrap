# ForgeOps Bootstrap — Project Specification

**Purpose**

ForgeOps Bootstrap is a production-grade infrastructure repository designed to provision, configure, maintain, migrate, verify, and update Linux VPS servers using a single, repeatable workflow.

This repository is the permanent foundation for all future VPS deployments. It is never a demo, tutorial, proof of concept, or experimental project. Every component is suitable for long-term production use.

This document covers two things, in order: the infrastructure layer below (complete, frozen, described in full detail), and the platform being built on top of it (see "Platform Vision" near the end — vision only, not yet implemented).

## Role

Built and maintained as if by a team consisting of: Principal Platform Engineer, Senior DevOps Engineer, Linux Systems Engineer, Docker Expert, Site Reliability Engineer, Security Engineer, Infrastructure Architect, Bash Engineer, Python Engineer.

Maintainability over shortcuts. No placeholder implementations. No TODO comments in place of functionality.

## Target Platform

- OS: Ubuntu 24.04 LTS
- Provider: OVH VPS
- Architecture: x86_64
- Infrastructure: Docker-first

## Primary Entry Points

| Script | Purpose |
| --- | --- |
| `./install.sh` | Installs and configures the complete server (idempotent, resumable) |
| `./migrate.sh` | Migrates an existing VPS into this environment over SSH |
| `./verify.sh` | Validates every installed component, generates a health report |
| `./update.sh` | Safely updates the complete infrastructure to pinned versions |
| `./uninstall.sh` | Safely removes installed components (never deletes user data without `--purge-data` + confirmation) |

## Versioning & Pinning Policy

Every installed component has an explicit version policy — "always latest" is not used for anything beyond OS security patches.

- Ubuntu OS packages: `unattended-upgrades` for security patches only; no silent LTS jumps.
- Docker Engine/Compose, Caddy, PostgreSQL, Redis, Portainer, Uptime Kuma, Watchtower: pinned in `configs/versions.env`.
- Node.js: pinned LTS major line.
- uv: pinned release.
- `update.sh` only ever moves components to what's declared in `configs/versions.env` — bumping a version is a deliberate edit to that one file.

## Secrets Management

- All runtime secrets live in `.env` (git-ignored, mode 600), generated by `install.sh` from `.env.example` on first run with random values.
- `.env.example` is a template only — no real secrets.
- `docker-compose.yml` consumes secrets exclusively via `env_file: .env`.
- `verify.sh` checks `.env` exists, is mode 600, and has no empty required keys — never prints values.
- External secret managers (Vault, SOPS, KMS) are out of scope for v1; the env-var injection contract is designed so one can be added later without changing how services consume secrets.

## install.sh

Idempotent and resumable via a state file (`logs/.install-state`); re-running skips completed steps and resumes from any failure point. `install.sh --reset-state` forces a full re-run.

Automatically: updates/upgrades Ubuntu, configures locale/timezone, installs Docker + Compose, Git, Python, uv, Node.js LTS, build-essential, curl, wget, jq, ripgrep, fzf, tmux, btop, htop, tree, ncdu, rsync, Fail2Ban, UFW, Caddy, Portainer, PostgreSQL, Redis, Uptime Kuma, Watchtower; creates project directories; installs the daily backup timer (see Backup, below); configures automatic security updates, log rotation, SSH hardening (password auth only disabled once a key is verified present); detects KVM (informational only, logged — no KVM-dependent tooling installed in v1); generates a full installation report via `verify.sh`.

## migrate.sh

Two-phase safety model: `--sync` (repeatable, non-destructive rsync-based copy while source keeps running) and `--finalize` (one-time cutover: stops only the migrated containers on the source, does a final delta sync, verifies SHA-256 checksums, and only then starts the destination — aborting before starting the destination if verification fails). `--rollback` restarts what `--finalize` stopped on the source. The source host is never modified outside the explicit stop in `--finalize`, and is never deleted from or force-shut-down — it remains a valid rollback target until the operator decommissions it manually.

Supports: projects, Claude configuration, Docker configuration, Docker Compose files, optional SSH configuration, optional Docker volumes. Preserves permissions/ownership/timestamps, resumes interrupted transfers, verifies via checksum manifest (not just size/count), supports `--dry-run` and `--verbose`.

## verify.sh

Checks: OS, kernel, CPU, memory, disk, networking, Docker, Docker Compose, Git, Python, uv, Node.js, Claude CLI, Caddy, Portainer, Redis, PostgreSQL, Fail2Ban, firewall, Docker networks, Docker volumes, running containers, health endpoints, KVM availability, secrets integrity. Generates console, Markdown (`logs/verify-report.md`), and JSON (`logs/verify-report.json`) reports.

## update.sh

Updates Ubuntu packages, pulls and recreates containers at the versions pinned in `configs/versions.env`, prunes unused Docker resources, cleans the apt cache, rotates logs, and runs `verify.sh` as a final gate. Aborts on any critical failure (failed apt upgrade, failed image pull, failed recreate) before touching the next stage, so a failed update never leaves a partially-updated stack running silently.

## uninstall.sh

Distinguishes components (packages, containers, images, systemd units, Caddy config, UFW rules) from user data (Postgres/Redis/Portainer/Uptime Kuma volumes, `backups/`, `logs/`, the configured projects directory). Removes components by default; `--purge-data` plus an explicit typed confirmation is required to delete user data. `--dry-run` lists what would happen without deleting anything.

## Docker Architecture

Everything that reasonably can run inside Docker does: Docker Compose, named volumes, named networks, health checks, restart policies, environment variables for config, secrets sourced from outside the repository via `.env`.

## Reverse Proxy

Caddy: automatic HTTPS and certificate renewal, compression, security headers, reverse proxy for every service that's opted into public exposure (see Service & Network Topology). Rendered from `templates/Caddyfile.template` by `scripts/render_caddyfile.sh` based on `.env`.

## Security

SSH hardening (password auth disabled only after key-based login is confirmed possible), UFW firewall, Fail2Ban, automatic security updates, least privilege, no hardcoded credentials, no default passwords.

## Backup

`scripts/backup.sh` runs daily via the `forgeops-backup.timer` systemd unit (installed by `install.sh`), dumping PostgreSQL (`pg_dump -Fc`) and Redis (RDB snapshot) plus critical config, archiving with a SHA-256 checksum manifest, and immediately verifying the archive — a backup that fails verification (bad checksum, unreadable Postgres dump) is deleted rather than kept. Retention is configurable via `BACKUP_RETENTION_DAYS` in `.env` (default 14 days). `scripts/restore.sh` re-verifies a chosen backup before touching anything, then requires a typed `restore` confirmation before overwriting live PostgreSQL/Redis data. See `ARCHITECTURE.md`'s Backups section for the full flow.

## Monitoring

Uptime Kuma, monitoring CPU, memory, disk, Docker, services, and health endpoints.

## Logging

Timestamped, structured, colored console output; dedicated `logs/` directory; weekly rotation with 8-week retention via `/etc/logrotate.d/forgeops`.

## Coding Standards

Shell: strict mode (`set -euo pipefail`), ShellCheck-clean, modular, no duplicated logic, meaningful function names, comments explaining intent (not restating code).

Python: type hints, `ruff` + `mypy` clean, modular, no duplicated logic, meaningful names, comments explaining intent.

## Error Handling

Every script exits with a meaningful status code, detects failures explicitly, displays readable errors, and supports rollback where the operation has one (`migrate.sh --rollback`, `update.sh`'s pre-update image snapshot for manual rollback).

## Future Extensions (architectural constraints, not v1 scope)

The v1 architecture (Docker-first, named networks/volumes, Caddy edge, `configs/versions.env` pinning) must not preclude, in future work: ForgeOps, FastAPI, Flask, AI services, Claude Code, Claude Cowork, MCP servers, Android virtual machines (when KVM is available), CI/CD, and development/staging/production environment separation. None of these are implemented, stubbed, or represented by placeholder code in v1.

## Platform Vision (ForgeOps) — vision, not v1 scope

Everything below this line describes where the repository is going, not what exists today. It does not change or supersede any rule above: the infrastructure layer (Docker/Caddy/Postgres/Redis/Portainer/Uptime Kuma, `install.sh`/`verify.sh`/`update.sh`/`migrate.sh`/`uninstall.sh`) is complete and frozen. Do not redesign or replace it except for a genuine production issue.

**Origin note:** this direction started as a narrower effort to cut Claude Code token usage (context compression / token budgeting). That's why "Context" is one of the eight layers below rather than the whole project — the platform grew out from that one problem, it wasn't planned top-down from an 8-layer diagram first.

### Mission

ForgeOps is not an AI assistant, chatbot, or IDE. It is a self-hosted backend that manages context, memory, projects, knowledge, and AI-client integrations, exposed primarily over the Model Context Protocol (MCP) so any compatible client — Claude Code, ChatGPT, Cursor, Continue, Gemini, Cline, future clients — can plug into the same persistent backend. The client is replaceable; the backend is the product.

### Principles

1. **Never reinvent mature OSS.** Integrate and configure existing self-hostable projects before writing custom code for a problem one already solves.
2. **One problem per component.** No service should try to be two layers at once.
3. **Git is the source of truth.** Config and state that can live in git, does.
4. **Docker Compose is mandatory** for every service — no bare-metal service processes.
5. **Every service is replaceable** — swapping the OSS project behind a layer must not require touching the other layers.
6. **No cloud dependency.** Everything must be able to run on a VPS, mini PC, NAS, or laptop with nothing but Docker.
7. **Evolution over rewrites.** Every change is a small, production-ready, reversible pull request — never a bulk rewrite. "No placeholders" applies *within* a shipped PR's stated scope, not as license to build all eight layers in one PR.

### The eight layers

| # | Layer | Responsibility | Status |
| - | ----- | --------------- | ------ |
| 1 | Infrastructure | Docker, networking, secrets, storage | **Complete, frozen** (this document, above) |
| 2 | Knowledge | Git, knowledge base, documentation, indexed/queryable project knowledge | Not started |
| 3 | Context | Context engine — compression, routing, token budgeting | Not started — this is the layer the project originally grew out of |
| 4 | Projects | Project/task manager, decisions log, archive | Not started |
| 5 | Memory | Long-term / semantic memory, knowledge graph | Not started |
| 6 | Integrations | MCP gateway + servers, GitHub, Google Drive, filesystem | **Next real PR** — see below |
| 7 | Observability | Logging, metrics, tracing | Not started |
| 8 | Clients | Claude Code, ChatGPT, Cursor, Continue, Cline, etc. — consumers of layers 1-7, not built by this repo | N/A (external) |

Priority order for build-out (highest first): **1. MCP ecosystem (Layer 6) → 2. Context (Layer 3) → 3. Memory (Layer 5) → 4. Knowledge (Layer 2) → 5. Client integrations (Layer 8, mostly external) → 6. Observability (Layer 7) → 7. Automation.** Layer 6 is first because every other layer is reached through it — nothing plugs in until there's a gateway to plug into.

### OSS candidate evaluation

Full research-backed evaluation of every candidate project considered per layer (what it is, self-hostability, maintenance signal, verdict, and why) lives in [`docs/OSS_EVALUATION.md`](docs/OSS_EVALUATION.md). Summary of confirmed integrate-candidates by layer:

- **Layer 6 (Integrations/MCP):** `IBM/mcp-context-forge` (primary MCP gateway candidate — registry, protocol translation, OTel, K8s-scalable) or `docker/mcp-gateway` (lighter alternative); Anthropic's `modelcontextprotocol/servers` as reference implementations to wrap, not production infra itself; Bifrost (`maximhq/bifrost`) as a separate LLM-provider gateway (distinct concern from the MCP-tool gateway — do not conflate the two).
- **Layer 5 (Memory):** Letta, Mem0, Cognee, and Graphiti all overlap here — pick one primary rather than running all four (see evaluation doc for the tradeoffs) — plus `gastownhall/beads` for git-backed agent task memory specifically.
- **Layer 2 (Knowledge):** Graphiti (temporal knowledge graph) is the strongest single candidate.
- **Layer 7 (Observability):** Langfuse — self-hostable via Docker Compose, purpose-built for LLM tracing/evals.
- **Automation (cross-cutting, not a numbered layer above but referenced in the mission):** LangGraph (orchestration), OpenHands (autonomous execution), n8n (workflow automation — note its Sustainable Use License is source-available, not OSI-permissive).
- **Layer 8 (Clients):** Cline is an actively-growing candidate to document support for; Continue is now archived/unmaintained post-acquisition and should not be treated as a dependency, only listed as a client ForgeOps still happens to support.
- **Rejected / needs verification before reuse:** "vexp" and "Context Mode" both target context-compression but their actual OSS licensing/maturity could not be confirmed from a public repo — verify before adopting either. "Claude Token Optimizer" is not one canonical project; if used, name the specific repo (e.g. `nadimtuhin/claude-token-optimizer`) rather than the generic phrase.

### Non-goals

ForgeOps does not become another ChatGPT clone, another web chat UI, another vector database, or another general-purpose workflow engine. It integrates mature OSS for each of those problems instead of building them from scratch.
